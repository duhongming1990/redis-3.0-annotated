    EXPIRE key ttl

### 9.4.2 保存过期时间

```cfml

    // 键的过期时间，字典的键为键，字典的值为过期事件 UNIX 时间戳

}
```

### 9.4.3 移除过期时间
    PERSIST key

### 9.4.4 计算并返回剩余生存时间
    TTL key
    PTTL key
    
### 9.4.5 过期键的判定
1. 检查给定键是否在过期字典中：存在，则获取。
2. 检查当前的UNIX时间戳与键的过期时间。

## 9.5 过期键删除策略
    定时删除：不现实
    惰性删除：expireIfNeeded函数实现
    定期删除：activeExpireCycle函数实现

## 9.6 AOF、RDB和复制功能对过期键的处理
    执行SAVE、BGSAVE生成RDB文件和执行BGREWRITEAOF生成的AOF不会包含过期键
    主服务器删除一个过期键之后，会向所有从服务器发送一条DEL命令，从服务器即使发现过期键也不会自作主张地删除它。

## 10.2 自动间隔性保存

### 10.2.1 设置保存条件
```
#   In the example below the behaviour will be to save:
#   after 900 sec (15 min) if at least 1 key changed
#   after 300 sec (5 min) if at least 10 keys changed
#   after 60 sec if at least 10000 keys changed
save 900 1
save 300 10
save 60 10000
```

redis.h/redisServer
```cfml

struct redisServer {
    struct saveparam *saveparams;   /* Save points array for RDB */
    int saveparamslen;              /* Number of saving points */
}

// 服务器的保存条件（BGSAVE 自动执行的条件）
struct saveparam {

    // 多少秒之内
    time_t seconds;

    // 发生多少次修改
    int changes;

};

```

## 11.3 AOF重写

### 11.3.1 AOF文件重写的实现
通过读取当前数据库状态来实现的。

重写程序在处理Hash、List、Set、ZSet时，元素超过64个时用多条命令执行。
```cfml
#define REDIS_AOF_REWRITE_ITEMS_PER_CMD 64
```

### 11.3.2 AOF后台重写
1. 开启子进程，维护一个AOF重写缓冲区。
2. 子进程完成AOF文件创建工作后，将重写缓冲区追加到新AOF文件的末尾。
3. 原子地（atomic）覆盖现有的AOF文件。

# 13 客户端
通过使用由I/O多路复用技术实现的文件事件处理器，
Redis服务器使用单线程单进程的方式来处理命令请求，
并与多个客户端进行网络通信。
```cfml
struct redisServer {
    // 一个链表，保存了所有客户端状态结构
    list *clients;              /* List of active clients */
}
 ```

## 13.1 客户端属性
```cfml
/* With multiplexing we need to take per-client state.
 * Clients are taken in a liked list.
 *
 * 因为 I/O 复用的缘故，需要为每个客户端维持一个状态。
 *
 * 多个客户端状态被服务器用链表连接起来。
 */
typedef struct redisClient {

    // 13.1.1 套接字描述符
    //伪客户端（fake client,载入AOF文件、执行Lua脚本) = -1
    //普通客户端 > -1
    int fd;

    // 13.1.2 客户端的名字
    robj *name;             /* As set by CLIENT SETNAME */

    // 13.1.4 查询（输入）缓冲区
    sds querybuf;

    // 13.1.5 命令与命令参数
    // 参数数量
    int argc;
    // 参数对象数组
    robj **argv;

    // 13.1.6 命令实现函数
    // 记录被客户端执行的命令
    struct redisCommand *cmd, *lastcmd;

    // 13.1.9 时间
    // 创建客户端的时间
    time_t ctime;           /* Client creation time */
    // 客户端最后一次和服务器互动的时间
    time_t lastinteraction; /* time of the last interaction, used for timeout */
    // 客户端的输出缓冲区超过软性限制的时间
    time_t obuf_soft_limit_reached_time;
    
    // 13.1.3 客户端状态标志
    int flags;              /* REDIS_SLAVE | REDIS_MONITOR | REDIS_MULTI ... */

    // 13.1.8 身份验证
    // 当 requirepass 不为 NULL 时
    // 代表认证的状态
    // 0 代表未认证， 1 代表已认证
    int authenticated;      /* when requirepass is non-NULL */

    // 13.1.7 输出缓冲区
    /* Response buffer */
    // 回复偏移量
    int bufpos;
    // 回复缓冲区
    char buf[REDIS_REPLY_CHUNK_BYTES];

} redisClient;
```

## 13.2.2 关闭普通客户端
硬性限制(hard limit) 立即关闭客户端

软性限制(soft limit) 同时满足<soft limit> <soft seconds>，立即关闭客户端

```cfml
# normal -> normal clients including MONITOR clients
# slave  -> slave clients
# pubsub -> clients subscribed to at least one pubsub channel or pattern
#
# The syntax of every client-output-buffer-limit directive is the following:
#
# client-output-buffer-limit <class> <hard limit> <soft limit> <soft seconds>
client-output-buffer-limit normal 0 0 0
client-output-buffer-limit slave 256mb 64mb 60
client-output-buffer-limit pubsub 32mb 8mb 60
```

## 13.2.3 Lua脚本的伪客户端

```cfml
   lua_State *lua; /* The Lua interpreter. We use just one for all clients */

 ```
